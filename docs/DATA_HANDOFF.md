# Grand Rapids Crime Animation — Data Handoff

> ⚠️ **Historical document.** This is the original spec for replacing an early
> *preview shell's* placeholder data with real, sourced data. It is kept for
> provenance of the project's origin. **The shipped project uses 100% real,
> sourced data — nothing on screen is simulated.** The "simulated data" below
> refers only to the throwaway preview shell that predated the real pipeline.
> See [`README.md`](../README.md) and [`PROVENANCE.md`](../data/grand-rapids-mi/PROVENANCE.md)
> for what actually ships.

A spec for replacing the shell's simulated data with real, sourced data. Hand this, plus `gr_crime_timeline.html`, to Claude Code.

## 1. Status

- **The shell works** (renders a real map, animated colored points, evolving heat, growing total chart, multiple map styles).
- **All data is simulated.** It is procedurally generated by a seeded random model (`makeSimulatedGR()`), not from any police record. The trends, neighborhood densities, and counts are invented to look plausible.
- The shell already exposes a clean data interface, so swapping in real data is a contained job — it does not require touching the rendering code.

## 2. Is the shell "ready"?

Ready:
- Map base + 4 styles (Dark, Light, Streets, Satellite), points by category, translucent heat, timeline scrubber, play/speed, growing cumulative chart, dispatch feed, auto-hiding controls, keyboard shortcuts.
- A documented data contract (`sourceFromIncidents`) that auto-derives the timeline span, map extent, legend, counters, and trend from whatever data you feed it.
- Hardened ingestion: bad rows are dropped, unknown categories bucket to a fallback, empty input throws a clear error.

Not done yet (the real-data job):
- Real data is not wired in.
- The 2000→present time span is currently only possible because the data is fake (see the hard limitation in §6).
- No rate normalization (crimes per capita), no attribution/credit text for the data source on screen, no automated fetch/refresh.

## 3. The data contract

The shell reads everything from one `source` object. The only line to change is the active-source selection:

```js
const source = makeSimulatedGR();   // <-- replace this
```

Replace it with:

```js
const source = sourceFromIncidents(rows, {
  title: "Grand Rapids · MI — Reported Incidents",
  categories: {
    persons:  { label: "Persons",  color: "#ff2e63" },
    property: { label: "Property", color: "#ffc233" },
    society:  { label: "Society",  color: "#34e0e0" }
  },
  fallbackCat: "property"   // unknown categories land here
});
```

Where `rows` is a flat array, one object per incident:

```js
{ date: "2025-11-04",   // Date or "YYYY-MM-DD" (incident date)
  lat: 42.9637,         // number (decimal degrees)
  lng: -85.6681,        // number
  cat: "persons",       // one of the category keys above
  type: "Assault",      // optional, shown in the dispatch feed
  place: "Heartside" }  // optional neighborhood/label
```

From this, the shell auto-computes the date range, map bounds/zoom, legend, counters, and the trend chart. No other code changes are required.

## 4. Category mapping (NIBRS → shell keys)

GRPD already classifies incidents into the three NIBRS groups, so the mapping is nearly 1:1:

- **persons** ← Crimes Against Persons: assault, homicide, kidnapping/abduction, sex offenses.
- **property** ← Crimes Against Property: arson, burglary/B&E, larceny/theft, motor vehicle theft, vandalism, fraud, forgery, embezzlement, extortion, robbery, stolen property.
- **society** ← Crimes Against Society: drug/narcotic, weapons violations, prostitution, pornography/obscenity, animal cruelty.

If a feed gives raw NIBRS offense codes instead of the group, map the offense code to its NIBRS group first, then to these keys.

## 5. Reliable data sources

Ranked by usefulness for this animation.

**A. GRPD open data (incident-level points) — best for real dots**
- Hub: `https://grpd-grandrapids.hub.arcgis.com/`
- Mapping: `crimemapping.com` (linked from the hub)
- Gives: block-level incident points (not exact addresses), NIBRS category, recent window.
- Limits: only the **most recent ~6 months**, updated monthly, ~1-month lag.
- Access: the ArcGIS Hub usually exposes a FeatureServer; query its REST endpoint and export GeoJSON (`...FeatureServer/0/query?where=1=1&outFields=*&f=geojson`). Claude Code should inspect the hub to find the exact layer URL.

**B. City of Grand Rapids open data portal**
- `https://grdata-grandrapids.opendata.arcgis.com/` and `https://www.grandrapidsmi.gov/GRData/Police-Data`
- Gives: official datasets and dashboards, downloadable as CSV/GeoJSON via ArcGIS; good place to confirm the canonical police layer.

**C. FBI Crime Data Explorer (CDE) — historical agency counts**
- `https://cde.ucr.cjis.gov/` (API documented via the FBI CDE / `api.data.gov` key).
- Gives: agency-level **annual/monthly counts** by offense for GRPD (look up GRPD's ORI on the CDE — do not guess it). UCR summary data goes back decades; NIBRS incident-level nationally is the standard from 2021.
- Limits: **counts only, no coordinates.** Public domain (CC0).

**D. Michigan State Police — MICR / "Crime in Michigan"**
- `https://www.michigan.gov/msp` (Crime in Michigan annual reports).
- Gives: agency-level historical counts for GRPD, statewide context. Aggregate, no coordinates.

**E. Direct data request / FOIA to GRPD**
- `https://www.grandrapidsmi.gov/Services/View-a-Grand-Rapids-Crime-Map` and the city data-request process.
- Gives: potentially historical incident-level records on request. May involve fees and may not include coordinates.

## 6. The hard limitation (read before promising 2000→now)

Public **incident-level data with coordinates exists only for ~the last 6 months** (source A). Going back to 2000, only **aggregate counts** are available (sources C/D) — no per-incident coordinates. So a true dot-by-dot 2000→present animation is not possible from public data. Three realistic options:

1. **Recent & real (recommended first):** use source A for the last ~6 months. Fully accurate points + heat; short time span. Set the shell's span to that window.
2. **Long & approximate:** drive the heat and the total chart from real aggregate counts (sources C/D, 2000→now), and synthesize point *positions* within neighborhoods to match each period's real totals. Honest only if the on-screen "Simulated/approximate placement" badge stays. Good for a dramatic long sweep.
3. **Long & real:** FOIA GRPD (source E) for historical incident-level data with coordinates. Most accurate long animation; depends on what GRPD will release and at what cost.

For monetized video: keep a visible data-source credit, and if any positions are synthesized, keep the "approximate" badge — don't present modeled data as exact records.

## 7. Tuning knobs for the video (locations in `gr_crime_timeline.html`)

- Layer switches: `const SHOW_POINTS`, `const SHOW_HEAT`.
- Default map style: `setBase("dark")` — change to `light`/`streets`/`satellite`.
- Heat look: the `L.heatLayer([], { radius, blur, minOpacity, max, gradient })` call; overall heat transparency is the CSS rule `.leaflet-heatmap-layer{opacity:.5}`.
- Point look: `makeDot()` — `radius`, `color` (outline), `fillColor`, `fillOpacity`.
- Point lifetime / heat memory: `const DOT_LIFE`, `const HEAT_WIN`.
- Default playback speed and presets: `let spd=30` and the `SPEEDS` array.
