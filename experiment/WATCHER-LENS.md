# watcher solutions playbook (grows every resolution)

The note-watcher reads this BEFORE resolving (known categories first) and
appends AFTER each resolution: the category, the ROOT CAUSE (not the symptom),
how wide the category sweep went, and the lesson. This file is the "getting
better at solving" half of the critic↔watcher ratchet. Rule of the house:
surface-patching one instance of a systemic issue is a failure mode — fix the
producing layer (template/engine/pipeline/routine) and sweep every instance.

- video-page-no-wayfinding (operator lands on a video page, doesn't know what to do / what "review" or "verify" means / how to publish / how to raise the score) | root cause: the detail view was an INFORMATION display (8 cryptic stage lights + confidence + meta + a feedback box) with no stage-aware "what's my next move?" instruction; the flow rules (Verify is a manual light flipped by Approve, Publish needs all-green, confidence is advisory-not-a-gate) lived only in server.mjs + producerNotes, never on screen | sweep: fixed at the ENGINE layer (pipeline/dashboard/index.html renderGuide) so ONE change covers ALL 20 video pages — a plain-language "Next step" guide: Step N/8 + one-line action + explanation + the full journey as a who-does-what checklist, stage-branched (automated stages="runs automatically, leave a note"; review=defines review + ✓Approve→Verify; all-green=⤴Publish CTA; published=▶Watch); relabeled the Approve button "Approve — turn on Verify"; surfaced Confidence as advisory with blockers as the to-do that lifts it. Verified in-browser + DOM across all 8 branches | lesson: a multi-stage operator tool must TELL the user their single next action at each state, not just DISPLAY status and assume they can decode it — wayfinding is content, not chrome. When a note says "applies to all videos", the fix belongs in the shared render path (index.html/engine), which makes the category sweep automatic. Owner-facing flow rules must be explained inline where the action happens, never left implicit in server code.

- owner-approve-leaves-satisfied-blocker-stale | root cause: confidence.json `blockers` is hand-maintained free text with NO code path to strip a blocker when its condition is met — so a re-render blocker kept open "pending owner re-approval" stays listed even after the re-render lands AND the owner re-approves (the operations-critic 15:13 note is the systemic version of this) | sweep: reconciled ONLY the one blocker causally gated on this approve (Boston's re-render blocker — producer explicitly wrote "verify light owner-gated pending re-approval"); did NOT touch the watch-through blocker (approve ≠ audio watch-through) nor other cities' ledgers (that broader auto-clear is the open critic note's job) | lesson: when resolving an owner APPROVE, don't just log evidence — check whether that approve was the GATING EVENT the producer set up for an open blocker, and if the artifact condition is also met, strip it (the ledger otherwise drifts and inflates priorityOf). Honesty guard: only clear a blocker whose condition is actually satisfied AND that this approve was gating; leave genuinely-separate advisory blockers (audio watch-through) open. Score must NOT bump on a re-approval of an already-counted design gain — that would double-inflate.

- config-field-missing-not-empty (a publish/dashboard feature depends on a youtube.json/config field the authoring step doesn't guarantee) | root cause: the producer's title-authoring step lands youtube.json but never guarantees `titleOptions`, so 3 late configs (denver-co/detroit-mi/milwaukee-wi) shipped with the field ABSENT while 17 carried 2 — the publish modal's title picker then had no alternates | sweep: authored exactly 2 alternates for ALL 3 gap cities (decline-% hook + raw-endpoint hook), built ONLY from producer-verified hook/punchline figures, each ≤100 chars, none crossing the FBI↔incident measurement seam; verified vs each config + confidence.json; SURGICAL text insert after the `title` key (json.dump re-serialize was reverted first because it expanded the compact `chapters` objects → 96-line churn) | lesson: (1) content that must be honest+per-city (verified alternate titles) can't be auto-generated by a template, but the authoring routine CAN be made to REQUIRE the field so the gap is caught at author time — fix filed to HARNESS 'requested'; (2) never re-serialize a hand-formatted JSON to add one key — `json.dump(indent=2)` reflows every compact sub-object; do a surgical text insert and diff-check (expect ~4 insertions, not ~90); (3) a titleOptions/superlative honesty check: an alternate title must never state a % or endpoint pair that spans the source seam — lead on the same-measure decline only.
