# Design the first live Q&A and Project Update #2

The first live Q&A should be a public checkpoint, not a subscriber-count stunt.
Its purpose is to surface unresolved questions, show what feedback changed, and
publish the next revision of the project.

## What is settled

- The Q&A happens after the public launch, not before production begins.
- Project Update #2 should record accepted changes and remaining disagreements.
- Subscriber count may be reported as context but is not enough by itself.

## Current proposal

Trigger the event after:

- two human-approved pilot releases;
- 30 substantive public contributions; and
- at least three expertise lanes represented.

This threshold is a draft.

## Three questions

1. Is this trigger large enough to produce a useful checkpoint without delaying
   it indefinitely?
2. Which three questions must the first Q&A answer?
3. What evidence should Project Update #2 publish so viewers can verify that
   participation changed the project?
